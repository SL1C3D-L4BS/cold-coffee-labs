## I. The bet

Greywater_Engine is a bet that a small, disciplined studio can build and ship a game engine in 2026–2028 that does things the majors cannot or will not do — **not** by out-featuring them, but by choosing four specific architectural commitments that compound into a differentiated product.

The four commitments are:

1. **A native, first-class AI copilot (BLD) baked into the engine core** as a Rust crate, with authoritative tool-use over the engine's public C++ API and integrated with the editor's undo stack.
2. **A text-IR-native visual scripting system** where a typed text IR is the serialized ground-truth and the node graph is a reversible projection — the representation that is easiest for AI to manipulate is also the one that diffs, merges, and survives refactoring.
3. **A Clang-only, reproducible build** on Windows and Linux from the first commit, with CMake + Corrosion bridging C++ and Rust as a single build graph.
4. **Vulkan 1.3 as a baseline feature**, not a minimum — dynamic rendering, timeline semaphores, bindless descriptors, and a forward+ clustered renderer designed for 2026 GPUs from day one.

No other engine in 2026 combines these. Each in isolation exists somewhere. **The combination is Greywater's moat.**

---

## II. Why this, and why now

The game-engine market in 2026 is consolidating around a handful of large commercial products plus a long tail of studio-internal engines. Indie and mid-tier studios overwhelmingly use the commercial options.

**What the market is missing:**

- **An engine that treats an AI copilot as a primary architectural concern rather than a marketing bolt-on.** Existing AI-assist tools for the commercial engines layer AI onto editors that were not designed for it. The layering is visible in every seam — tool surfaces lag the engine API, undo integration is incomplete, the "agent" runs in a separate process with JSON serialization you can feel.
- **A visual scripting system where text is canonical.** The dominant visual scripting systems made the graph ground-truth and pay the price in diffs, merges, and AI friction. Greywater inverts this — text IR is canonical, graph is a projection.
- **A modern, reproducible, cross-platform build on native compilers.** "Hello world on a new machine" is an adventure on most engines. Greywater's commitment: one `cmake --preset` invocation on a fresh clone on either OS, guaranteed reproducible.
- **A product built for a team of five to fifty.** The commercial engines are optimized for teams of hundreds. Every decision they make compounds against small-team efficiency. Greywater is sized for the studio we are.
- **An engine targeting mid-range hardware as the baseline, not the low-preset afterthought.** The RX 580 is our floor. Every Tier A feature runs there at 1080p / 60 FPS.

The window for this opportunity is **right now**. The Model Context Protocol stabilized in late 2025 and is the emerging standard for agentic tool surfaces. The Rust ML ecosystem reached production quality in 2024–2026. Vulkan 1.2 achieved broad driver support years ago. The dependency tree we need is mature simultaneously for the first time.

## III. The product

Greywater_Engine is one coherent product composed of **four linkable artifacts** plus **one hot-reloadable module**:

1. **`greywater_core`** — a C++23 static library. The engine. 22 subsystems covering renderer, ECS, audio, physics, animation, input, UI, networking, persistence, telemetry, cinematics, and more. It is not an executable; it is consumed by the four that follow.
2. **`greywater_editor`** — the authoring tool. Native C++ executable. Statically links `greywater_core` and links the Rust BLD crate via a narrow C-ABI. Uses Dear ImGui with docking and multi-viewport for all tooling UI.
3. **`sandbox`** — the engine's own feature test bed. Proves subsystems in isolation, without editor or game weight.
4. **`game_runtime`** — the shipping game executable. Lightweight; loads the gameplay module at startup.
5. **`gameplay_module`** — a hot-reloadable C++ dylib. The only dynamic library in the system. Change gameplay code, recompile, and the running process picks up the new module without shutdown.

**BLD (Brewed Logic Directive)** is the native AI copilot, implemented as a Rust crate inside `bld/`. It is linked into the editor (and into any other executable that wants an embedded agent — sandbox, runtime debug builds, CI tools). It is an MCP server; the editor chat panel is one client among many.

Platforms at v1: Windows (x64) + Linux (x64). macOS, consoles, and mobile are post-v1.

## IV. The architectural thesis

Greywater's seven structural commitments — the specific design decisions that make "baked-in AI" and "text-IR scripting" and "reproducible Clang build" more than marketing words:

1. **The engine's public C++ API is the agent's tool surface.** Tools are generated at compile time — `#[bld_tool]` proc macros on the Rust side, with callbacks into a stable C-ABI on the C++ side. No hand-written JSON. No translation-layer drift.
2. **The undo stack is the agent's commit log.** Every mutating tool produces an `ICommand` on the editor's command stack. Ctrl+Z reverses the agent's moves identically to the user's. Transaction brackets collapse multi-step agent actions into a single atomic undo.
3. **MCP is the wire from day one.** Spec version `2025-11-25`. The engine is a server; the editor is a client; so is everyone else — Claude Code, Cursor, Zed, CI automation, future agents not yet specified.
4. **BLD is embeddable, not editor-only.** A shipping game can embed the copilot for in-game modding tools, debug consoles, and QA automation — because the agent is a linkable static library, not locked to the editor process.
5. **HITL approval is protocol-native.** The MCP `elicitation` primitive drives the approval UI. Any compliant client renders it. No bespoke modal.
6. **Tool surface auto-synchronizes with the engine.** Adding or editing a `#[bld_tool]` function rebuilds the tool registry on the next `cargo build`. No drift, no gaps.
7. **The agent stress-tests the public API every day.** If BLD cannot accomplish a task, the C-ABI callback surface is missing an entry — the engine's exposed API is incomplete. This forcing function produces a materially cleaner API than any competing engine.

## VII. Risk analysis

The risks that can sink this project, in expected descending order:

### 1. Small-team burnout (high likelihood, catastrophic impact)

The single most likely cause of project failure. Mitigated **structurally** — not by willpower:

- Kanban WIP limit of 2, enforced.
- Daily log as ritualized memory; no day starts without reading yesterday's.
- 30-minute Sunday retros, no coding allowed.
- Two mandatory rest days per month.
- 732 pre-generated daily logs at project start — a contract with oneself written before fatigue accumulates.

### 2. Scope creep (high likelihood, high impact)

Mitigated by Tier A/B/C classification on every subsystem (`02_SYSTEMS_AND_SIMULATIONS.md`). Every Kanban card carries its tier. Every weekly retro asks: "is this still Tier A?" If not, it is reclassified or removed from scope — no ambiguous in-between.

### 3. Rust/C++ FFI tax (medium likelihood, bounded impact)

The FFI boundary runs through BLD's tool-call path, which is low-frequency and already JSON-serialized. The boundary is narrow (opaque handles + POD messages), covered by Miri nightly. In the worst case, we fork BLD's language in ~4 weeks.

### 4. Hardware / driver divergence on Vulkan (medium likelihood, medium impact)

Validation layers on in debug. Reference hardware list published. RenderDoc captures archived per milestone. Capability flags gate optional features. Deterministic cook output required across platforms.

### 5. Solo-dev → small-team transition (unknown likelihood, high impact)

At some point the project hires. The canonical docs (`00`, `01`, `architecture/*`, `CLAUDE.md`) are the onboarding system. An engineer who reads the canonical set in order should be productive in a week.

Full risk matrix with mitigations: `architecture/grey-water-engine-blueprint.md` §6 — 21 named risks, each with a concrete mitigation.

## IX. Success criteria

Greywater ships to LTS branch on 2029-05-28 (week 162) if and only if:

- Board-room demo runs unattended for ten minutes on reference hardware.
- Sponza renders at ≥ 144 FPS @ 1440p with forward+ lighting.
- BLD performs a full session end-to-end (scene → components → code → build → reload → verify) through the chat panel, with every action on the command stack.
- Two-client-plus-server replicated multiplayer runs 10 minutes at 60 Hz under 120 ms RTT with < 1 % desync.
- WCAG 2.2 AA accessibility audit passes on the sample HUD.
- Installer is signed and packaged for Windows (MSIX) and Linux (AppImage + DEB + RPM).
- Zero sanitizer warnings, zero Miri UB, zero clippy-deny across the full build.
- Engineering handbook is indexable by BLD and returns correct answers to 95 %+ of a canned question set.
