## 2. P0 findings — correctness / undefined behavior

### P0-1 — `MultiQueueScheduler::submit` stores dangling pointers
- **File:** `engine/render/frame_graph/submit.cpp:174-217`
- **Rule:** Vulkan 1.3 spec — valid usage of `VkSubmitInfo2`: `pWaitSemaphoreInfos`, `pSignalSemaphoreInfos`, `pCommandBufferInfos` must point to memory that stays live until `vkQueueSubmit2` returns. [VK_KHR_synchronization2](https://registry.khronos.org/vulkan/specs/1.3-extensions/man/html/VK_KHR_synchronization2.html)
- **What's broken:** The per-iteration `wait_infos` / `signal_infos` `std::vector`s and the per-iteration `cmd_submit_info` are stack-local to the inner `for (submission : submit_info.submissions)` loop. Their `.data()` pointers get stored into a `VkSubmitInfo2` that is then pushed into `submit_infos[queue_type]`. By the time `vkQueueSubmit2` is called (outer loop), the source vectors have been destroyed — we're reading freed stack memory.
- **Why it matters:** Random validation-layer crashes, GPU hangs, memory corruption. Will become live failure when the multi-queue path exercises (today's tests submit through the graphics path only).
- **Fix:** Hoist `wait_infos`, `signal_infos`, and `cmd_submit_infos` to outer-scope containers indexed by submission index, build all `VkSubmitInfo2`s referring into those stable containers, then call `vkQueueSubmit2` while the containers are still alive. Same scope, same frame.
- **Status:** RESOLVED (2026-04-20 evening, working tree). Complexity: M.

### P0-2 — Timeline-semaphore `get_next_value` is a side-effecting counter bump used as a read
- **File:** `engine/render/frame_graph/submit.cpp:40-42`, `:280-287`, `:353-355`
- **Rule:** Vulkan timeline semaphores — [Vulkan spec, "Timeline Semaphore"](https://registry.khronos.org/vulkan/specs/1.3-extensions/html/). A wait payload must equal a value that was (or will be) signalled; arbitrarily advancing the counter to pick a wait value creates waits that will never complete.
- **What's broken:** `get_next_value` does `return ++current_values_[queue_type]`. `build_submissions` calls it *twice* per pass (once for wait, once for signal), so every pass consumes two counter slots. `add_cross_queue_dependencies` calls it a third time per cross-queue dependency, advancing the counter without any matching signal.
- **Why it matters:** Deadlock or out-of-order execution whenever cross-queue dependencies fire. Masked today because the test harness only builds single-queue graphs.
- **Fix:** Split the API: `current_value(queue_type)` (pure read, used for waits) and `reserve_next_signal_value(queue_type)` (mutating, used only when a signal is actually emitted). Store per-submission `(wait_value, signal_value)` at build time, then use `(producer.signal_value, consumer.wait_value == producer.signal_value)` when adding cross-queue edges.
- **Status:** RESOLVED (2026-04-20 evening, working tree). `get_next_value` kept as `[[deprecated]]` for one cycle. Complexity: M.

### P0-3 — Swapchain `maxImageCount == 0` → zero images requested
- **File:** `engine/render/hal/vulkan_swapchain.cpp:74-76`
- **Rule:** Vulkan WSI — `VkSurfaceCapabilitiesKHR::maxImageCount == 0` means "no maximum except memory", not "zero maximum". [VkSurfaceCapabilitiesKHR](https://registry.khronos.org/vulkan/specs/1.3-extensions/man/html/VkSurfaceCapabilitiesKHR.html)
- **What's broken:** `std::min(capabilities.maxImageCount, std::max(3u, capabilities.minImageCount))` returns `0` whenever the driver reports `maxImageCount == 0`. On that code path `vkCreateSwapchainKHR` receives `minImageCount = 0` and the call fails with `VK_ERROR_INITIALIZATION_FAILED`.
- **Why it matters:** On a driver that reports unlimited maximum (common with modern AMD/NVIDIA), swapchain creation deterministically fails.
- **Fix:**
  ```cpp
  uint32_t desired = std::max(3u, capabilities.minImageCount + 1);
  uint32_t image_count = (capabilities.maxImageCount == 0)
      ? desired
      : std::min(desired, capabilities.maxImageCount);
  ```
- **Status:** RESOLVED (2026-04-20 evening, working tree). Complexity: S.

### P0-4 — `VulkanMemoryManager` creates VMA with incomplete `VmaAllocatorCreateInfo`
- **File:** `engine/render/hal/vulkan_memory.cpp:16-24`
- **Rule:** [VMA `VmaAllocatorCreateInfo`](https://gpuopen-librariesandsdks.github.io/VulkanMemoryAllocator/html/struct_vma_allocator_create_info.html) requires `physicalDevice` and `instance`. When the application uses volk (as we do), `pVulkanFunctions` must also be filled because VMA cannot find the entry points.
- **What's broken:** Only `device` and `vulkanApiVersion` are set. `physicalDevice`, `instance`, and `pVulkanFunctions` are default-null.
- **Why it matters:** Any allocation through this manager returns `VK_ERROR_INITIALIZATION_FAILED` or, worse, crashes when VMA dereferences null function pointers. Saved today only because nothing calls `VulkanMemoryManager` — `VulkanDevice::init_vma()` owns the live allocator.
- **Fix:** Either (a) retire `VulkanMemoryManager` as dead code (preferred — single source of truth on `VulkanDevice`), or (b) make it mirror `VulkanDevice::init_vma()` exactly and require the caller to pass instance + physical device.
- **Status:** RESOLVED (2026-04-20 evening, working tree) — option (a). Files deleted: `engine/render/hal/vulkan_memory.{hpp,cpp}` and the dead duplicate `engine/render/hal/CMakeLists.txt`. Complexity: S.

### P0-5 — Duplicate `ImGuizmo::BeginFrame()` per frame
- **Files:** `editor/app/editor_app.cpp:773` (inside `begin_frame`) AND `editor/panels/viewport_panel.cpp:122` (inside `on_imgui_render`)
- **Rule:** [ImGuizmo README](https://github.com/CedricGuillemet/ImGuizmo) — `BeginFrame()` is called **once** per ImGui frame.
- **What's broken:** Both call it. Second call resets gizmo hover/ID state after the editor application has already begun the frame.
- **Why it matters:** Intermittent gizmo hover glitches, broken drag-hysteresis, undefined hover state when multiple viewports dock.
- **Fix:** Keep the one call in `EditorApplication::begin_frame()` next to `ImGui::NewFrame()`; remove the call from `ViewportPanel::on_imgui_render()`.
- **Status:** RESOLVED (2026-04-20 evening, working tree). Complexity: S.

### P0-6 — `GizmoSystem::draw` hard-codes `SetOrthographic(false)`, overriding viewport camera mode
- **File:** `editor/viewport/gizmo_system.cpp:54`
- **Rule:** ImGuizmo: `SetOrthographic` must match the projection actually in use.
- **What's broken:** `ViewportPanel::on_imgui_render` correctly sets `SetOrthographic(camera_.mode() == CameraMode::Ortho)` on line 121, but `GizmoSystem::draw` unconditionally overrides that with `false` before `Manipulate`.
- **Why it matters:** Orthographic viewports render the gizmo with perspective math → wrong handle depth, wrong hit-testing.
- **Fix:** Thread the ortho flag into `GizmoSystem::draw(..., bool ortho)` OR remove the `SetOrthographic` call from `GizmoSystem` entirely (the viewport panel is already setting it).
- **Status:** RESOLVED (2026-04-20 evening, working tree) — option (a): flag threaded through `draw(...)` with default `false`. Complexity: S.

## 3. P1 findings — spec / policy risk

### P1-1 — HAL does not implement `RHICapabilities`; 1.3 features are unconditional
- **Files:** `engine/render/hal/vulkan_device.cpp`, `engine/render/frame_graph/frame_graph.cpp`
- **Rule:** `CLAUDE.md` non-negotiable #2 ("Vulkan 1.2 baseline; 1.3 opportunistic via `RHICapabilities`").
- **What's broken:** `VulkanDevice` unconditionally enables `VK_KHR_dynamic_rendering` + `VK_KHR_synchronization2` and always chains the feature structs. Frame graph always calls `vkCmdPipelineBarrier2`. There is no `RHICapabilities` symbol anywhere in `engine/render/hal/`.
- **Why it matters:** On any device that doesn't support these extensions, device creation will hard-fail rather than falling back. Our reference RX 580 supports both, so this is latent until cross-device testing.
- **Fix:** Introduce `engine/render/hal/capabilities.hpp` exposing `has_sync2()`, `has_dynamic_rendering()`, `api_version()`. Branch recording paths (legacy `vkCmdPipelineBarrier` + render passes vs sync2 + dynamic rendering) at the frame-graph level.
- **Status:** RESOLVED (2026-04-20 night + late-night amendment, working tree). `engine/render/hal/capabilities.{hpp,cpp}` added; `VulkanDevice` runs `probe_capabilities` early and enforces a Tier-A baseline of `{timeline_semaphores, descriptor_indexing, synchronization_2, dynamic_rendering}`. Device creation fails with a specific per-missing-feature error pointing at `docs/adr/0003-rhi-capabilities.md §2.1`. Frame graph drops its `has_sync2()` guard — sync2 presence is now a construction invariant, so the previous silent-skip-barriers-on-sync1 correctness hole is gone. `RHICapabilities` still surfaces Tier-B/Tier-G features (`mesh_shaders`, `ray_tracing_pipeline`, `ray_query`, `gpu_work_graphs`, `ext_memory_budget`) for future opportunistic branching. A real sync1 + legacy-render-pass backend becomes a Tier-G work item only when cross-hardware CI (lavapipe, MoltenVK) is added — it's no longer a lurking deferred obligation. Complexity: L.

### P1-2 — VMA not consuming `VK_EXT_memory_budget`
- **File:** `engine/render/hal/vulkan_device.cpp:173-188`
- **Rule:** [VMA — memory budget](https://gpuopen-librariesandsdks.github.io/VulkanMemoryAllocator/html/vk_ext_memory_budget.html)
- **What's broken:** `init_vma()` sets `ai.instance = VK_NULL_HANDLE`; `VMA_ALLOCATOR_CREATE_EXT_MEMORY_BUDGET_BIT` is not set; `VK_EXT_memory_budget` is not enumerated or enabled.
- **Why it matters:** We can't query budget on the RX 580's 8 GB for memory-pressure policy — which the game will need for Phase 20/21 planetary streaming.
- **Fix:** Enable `VK_EXT_memory_budget` (device extension), pass `instance` into `VmaAllocatorCreateInfo`, set the flag. Query budget at the beginning of each second and route it into the streaming scheduler.
- **Status:** UNRESOLVED. Complexity: M. Phase 20 hard dependency.

### P1-3 — Instance extension list names device-only extensions
- **File:** `engine/render/hal/vulkan_instance.cpp:28-33`
